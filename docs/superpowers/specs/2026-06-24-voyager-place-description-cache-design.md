# Shared Place-Description Cache — Design Spec

> Status: approved (brainstorm) · 2026-06-24 · branch `enrichment-cache`
> Builds on the merged `placeId` work (PR #1): stops carry a canonical Google
> `placeId`, which is the key this cache is built on.

## Design constraint — stay minimal

No background jobs, queues, cron sweepers, realtime infrastructure, or new
services. Everything is achieved with: one Postgres table (the cache), one small
audit table, one edge function (`enrich-place`), and client-side query caching.
Concurrency, stuck-generation recovery, validation, and staleness are all handled
**lazily on read / at generation time** — never a background worker.

## Context & goal

A stop's description — **Story (`history`)**, **Interesting Facts (`facts`)**,
**Experience (`tips`)** (plus legacy `notice`) — is currently generated by the AI
**per stop** (`generateStopDetail` in the browser) and stored inline. The same
place in two trips generates twice; there's no single place to maintain, audit, or
debug a description.

We want a **global, shared, server-controlled library** keyed by Google
`placeId`: generate once, reuse everywhere, **instant** reads, founder-controlled
**maintenance**, **sealed** (no user write path), and **debuggable** (we can tell
what place, which exact prompt, which model, and when).

## Core model — one shared, server-controlled source of truth

- A place's description lives in **one** global entry (`place_cache`), keyed by
  `placeId` + `prompt_version`. It is **not** copied onto the stop (for placeId
  stops).
- A placeId stop **displays whatever the library currently holds**. Change the
  entry once → every stop of that place, every trip, updates (on next load).
- The library is **server-only**: only the **`enrich-place`** edge function (service
  role) touches it. **No client write path.**
- **Typed-by-name stops** keep their own per-stop copy (today's behavior).

## Non-goals

- **Offline trips** — future **premium** tier (free tier reads live, cached).
- **No per-user "Generate / Re-generate" button** — regeneration is founder-only.
- **No per-stop storage for placeId stops** (existing fields are a transition
  fallback only).
- No admin "Places" UI in v1; no realtime; **no background workers** (see
  constraint).

## Architecture & data flow

```
DISPLAY a stop's description
  useStopDescription(stop):
    if stop.placeId → enrich-place:get (cached client-side)
        ready       → show it
        pending     → loading UX, poll briefly, then "still writing — check back"
        failed      → fallback to stop's legacy fields, else calm error
        unsupported → calm "no description" (terminal; place not found/unsuitable)
        miss        → enrich-place verifies + claims lock + generates → ready
    else (by-name)  → the stop's own stored fields (generated once, not cached)

GENERATE / WRITE (only inside enrich-place, service role)
  validate shape → VERIFY via Google Place Details (exists? canonical metadata)
    not found     → write a minimal `unsupported` row (negative cache), return
    exists        → claim row lock (atomic) → Wikipedia grounding + buildEnrichPrompt
                    + AI (ai-proxy, founder/credits-gated) + parseStopDetail
                    → ATOMIC write { content + full provenance + status='ready' }
                    → audit row → return
```

### The edge function: `supabase/functions/enrich-place`

The single authority for read/verify/generate/write/regenerate. The browser
`generateStopDetail` pipeline moves server-side here (pure helpers ported to Deno).
Actions:

- **`get`** `{ placeId, name?, destination?, coords?, placeTypes? }` →
  `{ status: 'ready'|'pending'|'failed'|'unsupported', content? }`. A `ready` hit
  does **no AI/Google work** (free, ungated). A miss verifies + generates (gated,
  validated, rate-limited, lock-protected). Client-passed fields are hints/
  fallback; **canonical metadata comes from Google Place Details** at generation.
- **`getBatch`** `{ placeIds }` → `{ placeId → content }` for **`ready` hits only**
  (no Google, no AI, no lock claims) — pre-warms a trip in one read.
- **`regenerate`** `{ placeId, force? }` → **founder-only** (`profile.role ===
  'founder'`, verified server-side). Forces a fresh generation and overwrites —
  **except** a `manual_lock` entry is preserved unless `force: true`. Audited.

Keys never appear in responses. Any failure returns `failed`/`unsupported`, never
a 5xx.

> **Server-side test caveat:** Deno isn't local, so the function's pure helpers
> (ported prompt builder/parser, validation, version + `prompt_id` hashing, and
> the lock/lease/cooldown decision logic) are unit-tested in the vitest app where
> authored; the deployed function is verified by a live probe. The client read
> layer + caching + hook + validation are fully vitest-tested.

## Place verification (no fake/junk places — required)

Regex shape alone is insufficient, so before any content row is generated:
1. **Shape check:** `placeId` matches `^[A-Za-z0-9_-]{16,}$` (cheap reject of
   obvious garbage; no row created).
2. **Existence check:** call **Google Place Details** (the same call
   `place-search` already uses) for the `placeId`.
   - **Not found** → write a minimal **`unsupported`** row (negative cache, no
     content) so we never re-look-up or re-attempt it; return `unsupported`. Only a
     founder `regenerate(force)` can revive it.
   - **Found** → proceed; the returned `displayName / formattedAddress / location /
     types / region` become the **authoritative metadata** stored on the row
     (client hints are ignored when Google answers).
   - **Transient lookup error** (network/5xx, not "not found") → treated as a
     generation **failure** (cooldown), not a terminal `unsupported`.

This is one Google call per *first* generation of a place (then cached forever),
so it's cheap and also removes any need to trust client-supplied metadata.

## Generation locking & states (stampede protection — required)

The cache row **is** the lock — no queue/worker. State machine:
`generation_status ∈ { generating, ready, failed, unsupported }` with
`generation_started_at`, `generation_attempts`, `generation_error`.

**On a `get` miss / non-ready row:**
1. **Claim atomically:** `INSERT (place_id, prompt_version, generation_status=
   'generating', generation_started_at=now(), generation_attempts=1) ON CONFLICT
   (place_id, prompt_version) DO NOTHING`.
   - **Won** → verify (above) then generate. On success, perform the **single
     atomic write** below. On error → `status='failed'`, `generation_error`,
     `attempts+1`.
   - **Conflicted** → read the existing row and branch:
2. **Existing-row branch:**
   - `ready` → return content.
   - `unsupported` → return `unsupported` (terminal; no retry on `get`).
   - `generating` within `LEASE_SECONDS` (≈60) → `pending`.
   - `generating` older than `LEASE_SECONDS` → **stuck lease** (owner died);
     reclaim via conditional update and generate. **Lazy** recovery — next read
     heals it; no sweeper.
   - `failed` within `FAILURE_COOLDOWN` (≈5 min) → `failed`.
   - `failed` past cooldown but `generation_attempts < MAX_ATTEMPTS` (≈5) → retry.
   - `generation_attempts ≥ MAX_ATTEMPTS` → only founder `regenerate` retries.

**Write atomicity (required):** a successful generation writes **content +
`prompt_version` + `prompt_id` + `model` + metadata + timestamps + `status='ready'`
in ONE statement/transaction.** A row can never be `ready` without matching
prompt/model provenance — no partial writes.

**Outcomes:** success = one AI call per `(placeId, version)`, concurrent callers
share the `ready` result; failure = `failed` + cooldown + capped retries;
abandoned = lease reclaim on next read; nonexistent/unsuitable = `unsupported`
(negative cache, founder-force only).

## Data model

### Table `place_cache` (global; operator SQL — no migration framework in repo)

| column | type | notes |
|---|---|---|
| `place_id` | `text` | Google place id — **PK part** |
| `prompt_version` | `int` | coarse version (current = 2) — **PK part**; drives version-bump regen |
| `prompt_id` | `text` | exact prompt provenance (hash of the prompt template) |
| `supersedes_version` | `int` | nullable — version this entry regenerated from (provenance) |
| `generation_status` | `text` | `generating` \| `ready` \| `failed` \| `unsupported` |
| `generation_started_at` | `timestamptz` | lease timestamp |
| `generation_attempts` | `int` | retry/abuse guard (default 0) |
| `generation_error` | `text` | nullable failure metadata |
| `content_source` | `text` | `generated` \| `manual` (provenance; default `generated`) |
| `manual_lock` | `boolean` | curated override — survives version bumps; **≤1 per place** (default `false`) |
| `is_stale` | `boolean` | maintenance flag (default `false`) |
| `history` | `text` | Story |
| `facts` | `jsonb` | Interesting Facts (string array) |
| `tips` | `text` | Experience |
| `notice` | `text` | legacy, nullable |
| `source` | `text` | Wikipedia extract used (provenance) |
| `place_name` | `text` | Google `displayName` |
| `address` | `text` | Google `formattedAddress` |
| `lat` / `lng` | `double precision` | Google `location` |
| `place_types` | `jsonb` | Google `types` |
| `country_code` | `text` | ISO-2, if known |
| `region` | `text` | state/region, if known |
| `model` | `text` | generation model id (e.g. `claude-opus-4-8`) |
| `generated_at` | `timestamptz` | first generation time |
| `updated_at` | `timestamptz` | last (re)generation / edit time |
| `last_verified_at` | `timestamptz` | last Google Place Details verification |
| `edited_by` | `text` | nullable — user id of a manual edit |

- **Primary key:** `(place_id, prompt_version)` — leading `place_id` serves the
  per-place reads; no extra hot-path index needed.
- **Partial unique index (required):** `UNIQUE (place_id) WHERE manual_lock` — at
  most **one** curated override per place, so the read precedence is unambiguous.
- **RLS:** deny all to `anon`/`authenticated`; **service role only**.

### Table `place_cache_audit` (lightweight operational log)

`id` (identity, PK), `place_id` (indexed), `prompt_version`, `action`
(`generate` | `regenerate` | `manual_edit` | `mark_stale`), `actor` (user id or
`system`), `model?`, `prompt_id?`, `detail?`, `created_at` (default `now()`).
Index `(place_id, created_at)`; service-role-only.

### Read precedence (manual override + version bumps)

`get` resolves a `place_id` by preferring, in order:
1. the single **`manual_lock = true`** row (≤1 by the partial unique index) — a
   **deliberate** curated override that **survives version bumps**; then
2. the **`prompt_version = CURRENT_ENRICH_VERSION`** row.

Only an explicitly **locked** entry escapes future prompt versions. (See
Manual-edit protection for why this replaces "manual-wins-forever".)

### Stop fields (already exist from PR #1 — additive)

`Stop.placeId?` (cache key); `Stop.history/facts/tips/notice?` kept but, for
placeId stops, no longer written by new code — transition fallback + by-name
storage only.

## Provenance & versioning

- `CURRENT_ENRICH_VERSION` = **2** (current "facts-separated" prompt; old browser
  prompt = **1**). Defined in app code (`enrich.ts`) for the client query key and
  **mirrored** in the function; bumping updates both (called out in the plan).
- `prompt_id` = stable hash of the **exact** prompt template string, stamped on
  every write — answers *"which exact prompt produced this?"* beyond the integer.
- `supersedes_version` records the prior version a regeneration replaced —
  provenance for debugging version-bump generations.
- **Version-bump behaviour (storm-safe):** bumping the integer makes places
  regenerate **lazily, only when next accessed** — never a platform-wide burst.
  Per-place generation is **lock-deduplicated, gated, and rate-limited**, so even
  at large scale only accessed places regenerate, spread over time. (Locked
  manual entries are skipped by precedence.)
- Every write stamps `model`, `place_name/address/lat/lng/place_types/country/
  region`, `source`, and timestamps — so months later we know **what / which
  prompt + model / when**.

## Manual-edit protection (deliberate override, not accidental opt-out)

The previous "manual always wins across every version" risked a minor typo fix
silently opting a place out of *all* future prompt improvements. Replaced with a
two-level model:

- **`content_source = 'manual'`** = provenance only ("a human last wrote this").
  A plain manual edit applies to **that version's** row and is **naturally
  superseded** when the version is bumped (so minor edits don't escape future
  improvements) — and may be overwritten by a founder `regenerate` (audited).
- **`manual_lock = true`** = a **deliberate curated override** the founder sets
  when content must persist. Only locked entries **win across versions** (read
  precedence) and are **protected from regeneration** unless `force: true` (which
  unlocks + regenerates, audited). At most one per place (partial unique index).

So: fix a typo → it lives at the current version and is replaced when prompts
improve. Hand-craft a definitive description → **lock it** and it persists. Nothing
is *silently* destroyed — every change is audited.

## Validation & abuse protection

- **Shape + existence** validation (see Place verification) — fake/nonexistent
  placeIds never create a content row; nonexistent → cheap `unsupported` negative
  cache (no repeated lookups).
- **Per-IP + per-user rate limit** on generation (`get` miss + `regenerate` +
  Place Details lookups), reusing the `place-photo` limiter. `getBatch` and `ready`
  hits aren't generation and aren't gated by it.
- **`generation_attempts` cap + `FAILURE_COOLDOWN`** stop failure storms; the
  **row lock** stops duplicate concurrent generations; `unsupported` stops retrying
  permanently-unsuitable places.
- The existing **founder/credits gate** (`ai-proxy`) gates all generation;
  `regenerate` additionally requires founder.

## Client read layer & performance (instant, not laggy)

`useStopDescription(stop)` → `{ history, facts, tips, notice, state }`:
- **placeId stop** → TanStack Query `['place-desc', placeId, version]` calling
  `enrich-place:get`. **In-memory cache** (instant after first fetch),
  **stale-while-revalidate** (edits propagate on next load without blocking),
  pre-warm on trip-open + on-add. `pending`/`failed`/`unsupported` drive Loading
  behavior.
- **by-name stop** → the stop's stored fields (generated once, no cache write).
- Read sites updated to use this hook: `Guide` (`StoryTabs`), `StopDetail`,
  `ArrivalView`.

> Optional (not core): persist the query cache to localStorage for instant reads
> across reloads — a device-local perf cache, distinct from premium offline.

## Pre-warm scaling

Bounded so large trips never fire a request per stop:
- Warm only the **active day's** stops (Guide) / **visible** stops (Plan).
- Cap `getBatch` to `PREWARM_BATCH_MAX` (≈25); overflow warms the first batch,
  rest lazy-load on view.
- `getBatch` returns **`ready` hits only** and never generates → one cheap read,
  no AI burst on trip open.

## Loading behavior (intentional, never blocking; UI poll ≠ backend lease)

- **Generating / pending:** a labeled shimmer — *"Writing this place's story…"* —
  and the query polls every ~1.5s, but **only for `UI_POLL_BUDGET` (≈12s)**, not
  the full backend lease. After the budget it switches to a calm *"Still writing
  this — check back in a moment"* and **stops polling** (re-checks on next view).
  So an Anthropic slowdown never traps the user watching a spinner for a minute;
  the backend lease can stay 60s independently.
- **Temporary failure:** show the stop's legacy fields if present, else a calm
  *"We couldn't write this place's story just now."* with a single quiet retry
  (respecting cooldown). No spinner spam, no error wall.
- **Unsupported:** a calm "no description for this place" — terminal, no retry UI.
- **Adding/viewing a stop is never blocked.**

## Auditability (no double-logging)

`place_cache_audit` records generate / regenerate / manual_edit / mark_stale with
actor + timestamp — operational visibility, nothing more:
- `enrich-place` writes **`generate`** and **`regenerate`** rows itself.
- A single `AFTER UPDATE` trigger on `place_cache` writes **`manual_edit`** **only
  when** `NEW.content_source = 'manual'` **and** content actually changed
  (`history/facts/tips IS DISTINCT FROM` old). Because `enrich-place` writes
  `content_source = 'generated'` on (re)generation, **the trigger never fires for
  a regenerate → no duplicate entry.** The trigger also writes **`mark_stale`**
  when `is_stale` flips `false→true`. No-op updates log nothing.

## Stale-content handling (simple; no auto-regen)

- `generated_at` / `updated_at` / `last_verified_at` make age visible; a founder
  may set `is_stale = true` (logged). **Stale entries are still served** (no UX
  impact) — just flagged for attention. No forced regeneration.
- **Cheap drift flag:** when a generation's Google Place Details metadata
  (name/address/coords) is **materially different** from what a prior entry for
  that place stored, the divergence is noted in the audit `detail` and other-
  version rows for that place may be set `is_stale = true` — a free signal that
  the underlying place changed, **not** a regeneration.
- A future time-based policy ("offer refresh past N years") is enabled by these
  columns but out of scope.

## Security

- `place_cache` + `place_cache_audit` are **service-role-only**; the app reaches
  descriptions solely through `enrich-place`. **No user write path.**
- **`ready` reads are free + ungated** (no AI/Google work).
- **Generation** is **shape+existence-validated**, **rate-limited** (per IP+user),
  **lock-protected**, and **gated** (founder/credits); `regenerate` requires
  founder. Google + Anthropic keys stay server-side.

## Failure & degradation

| Condition | Behaviour |
|---|---|
| `enrich-place` / network down | hook returns the stop's legacy fields if any, else a calm empty state; **never blocks** |
| Concurrent misses for one place | one generation (row lock); others `pending` then the shared result |
| Generation fails (transient) | `failed` + `generation_error`; cooldown + capped retries; founder force-retry |
| Place not found / unsuitable | `unsupported` (negative cache); founder force-retry only |
| Stuck `generating` (owner died) | reclaimed on next read after `LEASE_SECONDS` |
| Invalid/fake placeId | rejected by shape; nonexistent → cheap `unsupported`, no content row |
| Pre-warm batch fails | ignored — descriptions fetch lazily on view |

## Cost

Library hits eliminate repeat AI calls (free/ungated). Generation only on a true
miss or founder `regenerate` — verified, gated, rate-limited, lock-deduplicated.
`getBatch` never generates. One Google Place Details call per first generation
(then cached). Version bumps regenerate lazily per-access, never platform-wide.

   `AFTER UPDATE` manual-edit/mark-stale trigger.

## Acceptance criteria

- A place's description is generated **once** and reused across trips/users (a
  second add of the same `placeId` is a free, ungated `ready` hit).
- **Concurrent** misses → **exactly one** generation; others get `pending` then the
  shared result. A successful `ready` row **always** carries matching
  `prompt_id`/`model`/version (atomic write — no partial rows).
- A **stuck `generating`** row is reclaimed on next read after the lease; failures
  are cooled-down + capped; **nonexistent/unsuitable** places become terminal
  `unsupported` (negative cache), not retried.
- **Fake/nonexistent placeIds are verified out** via Google Place Details before
  any content row exists; generation is **rate-limited** (IP+user) and behind the
  **founder/credits gate**.
- A plain **manual edit** applies at its version and is **superseded by a version
  bump** (does not permanently opt the place out of improvements). Only a
  **`manual_lock`** entry survives version bumps and resists regeneration (unless
  `force`); **at most one locked entry per place**. Nothing is *silently*
  destroyed — all changes audited.
- Editing/regenerating a place updates **every** stop of that place across trips
  (on next load). Reads are **instant** after pre-warm/first fetch; pre-warm is
  **bounded** and triggers **no** generation.
- First-gen, pending, failure, and unsupported states render **intentional** UX,
  the **UI poll budget (~12s) is shorter than the backend lease (~60s)**, and
  **nothing blocks** adding/viewing a stop.
- Every entry records **place name/address/coords/types**, **model**, **prompt_id**
  + version (+ `supersedes_version`), and **generated/updated/verified**
  timestamps.
- Audit log captures **generate / regenerate / manual_edit / mark_stale** with
  actor + timestamp, and the manual-edit trigger **cannot double-log** during a
  regenerate.
- Old-prompt (v1) content is never served/written; version-bump regeneration works
  for generated entries and is **storm-safe** (lazy, per-access, lock+rate-limited).
- By-name stops still get a description (stored on the stop), without the library.
- **No background jobs, queues, cron, realtime, or new services.**

## Out of scope / future

- **Premium offline** (snapshot into trip data); **Admin "Places" page**;
  **realtime push**; **time-based staleness policy**; **shared cache for by-name
  stops**.
