# Shared Place-Description Cache — Design Spec

> Status: approved (brainstorm) · 2026-06-24 · branch `enrichment-cache`
> Builds on the merged `placeId` work (PR #1): stops carry a canonical Google
> `placeId`, which is the key this cache is built on.

## Design constraint — stay minimal

No background jobs, queues, cron sweepers, realtime infrastructure, or new
services. Everything is achieved with: one Postgres table (the cache), one small
audit table, one edge function (`enrich-place`), and client-side query caching.
Concurrency, stuck-generation recovery, and staleness are all handled **lazily on
read** using row state + timestamps — never a background worker.

## Context & goal

Today a stop's description — **Story (`history`)**, **Interesting Facts
(`facts`)**, **Experience (`tips`)** (plus legacy `notice`) — is generated by the
AI **per stop** (`generateStopDetail` runs in the browser) and stored inline on
the stop. The same place in two trips generates twice; a deleted-and-re-added
place regenerates; and there is no single place to maintain or audit a
description.

We want a **global, shared, server-controlled library** of descriptions keyed by
Google `placeId`: generate a place's description **once**, reuse it everywhere,
make reads **instant**, let a founder **maintain** it (edit/regenerate → updates
everywhere), and keep it **sealed** (no user write path) and **debuggable** (we
can tell what place, which prompt, which model, and when).

## Core model — one shared, server-controlled source of truth

- A place's description lives in **one** global library entry (`place_cache`),
  keyed by `placeId` + `prompt_version`. **It is not copied onto the stop** (for
  placeId stops).
- A stop with a `placeId` **displays whatever the library currently holds** for
  that place. Change the entry once → every stop of that place, in every trip,
  shows the new text (on its next load — see Performance).
- The library is **server-only**: the app never reads or writes the table
  directly. A single **edge function (`enrich-place`)** is the only thing that
  touches it (service role). There is **no client write path**.
- **Typed-by-name stops** (no `placeId`) keep their own per-stop copy (today's
  behavior), generated once and stored on the stop.

## Non-goals

- **Offline trips** — deferred to a future **premium** tier. Free tier reads live
  (cached for speed). Premium later snapshots descriptions into trip data.
- **No per-user "Generate / Re-generate" button** — regeneration is founder-only.
- **No per-stop description storage for placeId stops** — they read from the
  library. (Existing stored fields remain only as a transition fallback.)
- No full admin "Places" management UI in v1 (a minimal founder regenerate action
  suffices; the Supabase dashboard covers hand-edits).
- No realtime push (refetch-on-load propagates edits).
- No background workers of any kind (see Design constraint).

## Architecture & data flow

```
DISPLAY a stop's description
  useStopDescription(stop):
    if stop.placeId → enrich-place:get (cached client-side; see Performance)
                       ready → show it
                       pending (someone is generating) → loading UX + poll/retry
                       failed → fallback to the stop's legacy fields, else calm error
                       miss  → enrich-place claims the lock + generates → ready
    else (by-name)  → the stop's own stored history/facts/tips
                       (generated once via enrich-place, no cache write)

GENERATE / WRITE (only inside enrich-place, service role)
  validate place → claim row lock (atomic) → Wikipedia grounding +
  buildEnrichPrompt + AI (via ai-proxy, founder/credits-gated) + parseStopDetail
  → write content + provenance + status='ready' → audit row → return
```

### The edge function: `supabase/functions/enrich-place`

The single authority for reading, generating, writing, and regenerating
descriptions. The browser `generateStopDetail` pipeline (Wikipedia extract →
`buildEnrichPrompt` → AI → `parseStopDetail`) **moves server-side** here (pure
helpers ported to Deno). Actions:

- **`get`** `{ placeId, name, destination?, coords?, placeTypes? }` →
  `{ status: 'ready'|'pending'|'failed', content? }`. See **Generation states**.
  A `ready` hit does **no AI work** (free, ungated). A miss claims the lock and
  generates (gated + validated + rate-limited).
- **`getBatch`** `{ placeIds: [...] }` → `{ placeId → content }` for **ready hits
  only** (no generation, no lock claims) — pre-warms a trip in one round trip.
- **`regenerate`** `{ placeId, name, destination?, coords?, placeTypes?, force? }`
  → **founder-only** ("admin" = `profile.role === 'founder'`; the server verifies
  it). Forces a fresh generation and overwrites the entry — **except** a
  manually-edited entry is preserved unless `force: true` (see Manual-edit
  protection). Writes an audit row.

The function never returns the Google/Anthropic keys. Graceful: any failure
returns a `failed`/empty result, never a 5xx.

> **Server-side test caveat:** Deno isn't available locally, so the function's
> pure helpers (ported `buildEnrichPrompt`/`parseStopDetail`/validation/version +
> prompt-hash logic) are unit-tested in the Node/vitest app where they're
> authored; the deployed function is verified by a live probe. The client read
> layer + caching + hook + validation are fully unit-tested in vitest.

## Generation locking & states (stampede protection — required)

The cache row **is** the lock — no queue or worker. The row carries a small state
machine: `generation_status ∈ { generating, ready, failed }` plus
`generation_started_at`, `generation_attempts`, and `generation_error`.

**On a `get` miss / non-ready row, `enrich-place` does:**

1. **Claim atomically:** `INSERT (place_id, prompt_version, generation_status=
   'generating', generation_started_at=now(), generation_attempts=1) ON CONFLICT
   (place_id, prompt_version) DO NOTHING`.
   - **Insert won** → this request owns the generation. Generate → on success
     write content + provenance + `status='ready'`; on error write
     `status='failed'`, `generation_error`, bump `generation_attempts`.
   - **Insert conflicted** → someone else owns it (or it's ready/failed); read the
     existing row and branch below.
2. **Existing row branch:**
   - `ready` → return content.
   - `generating` **and** `generation_started_at` within `LEASE_SECONDS` (e.g. 60)
     → return `pending` (caller shows loading UX and retries).
   - `generating` **but** older than `LEASE_SECONDS` → **stuck/abandoned lease**
     (the owner died mid-generation). Reclaim with a conditional update
     (`SET generation_started_at=now(), generation_attempts=attempts+1 WHERE
     generation_status='generating' AND generation_started_at < now()-LEASE`) and
     generate. Recovery is **lazy** — the next request heals it; no sweeper.
   - `failed` **and** within `FAILURE_COOLDOWN` (e.g. 5 min) → return `failed`
     (caller shows the calm error / legacy fallback; does **not** hammer).
   - `failed` **past** the cooldown, or `generation_attempts` over `MAX_ATTEMPTS`
     (e.g. 5) → only a founder `regenerate` retries (prevents infinite retry
     abuse); a plain `get` returns `failed`.

**Outcomes documented:**
- **Success:** one AI call per `(placeId, prompt_version)`; concurrent callers get
  `pending` then the shared `ready` result — never duplicate generations.
- **Failure:** `status='failed'` + `generation_error`; served as `failed` with a
  cooldown; founder can force a retry.
- **Abandoned/stuck:** a `generating` lease older than `LEASE_SECONDS` is reclaimed
  on the next read.

## Data model

### Table `place_cache` (global; operator-created SQL — no migration framework in repo)

| column | type | notes |
|---|---|---|
| `place_id` | `text` | Google place id — **PK part** |
| `prompt_version` | `int` | coarse version (current = 2) — **PK part**, drives version-bump regen |
| `prompt_id` | `text` | exact prompt provenance (hash/identifier of the prompt template) |
| `generation_status` | `text` | `generating` \| `ready` \| `failed` |
| `generation_started_at` | `timestamptz` | lease timestamp |
| `generation_attempts` | `int` | retry/abuse guard (default 0) |
| `generation_error` | `text` | nullable failure metadata |
| `content_source` | `text` | `generated` \| `manual` (default `generated`) |
| `is_stale` | `boolean` | maintenance flag (default `false`) |
| `history` | `text` | Story |
| `facts` | `jsonb` | Interesting Facts (string array) |
| `tips` | `text` | Experience |
| `notice` | `text` | legacy, nullable |
| `source` | `text` | Wikipedia extract used (provenance) |
| `place_name` | `text` | resolved name (what was enriched) |
| `address` | `text` | formatted address |
| `lat` / `lng` | `double precision` | coordinates |
| `place_types` | `jsonb` | Google place types/category |
| `country_code` | `text` | ISO-2, if known |
| `region` | `text` | state/region, if known |
| `model` | `text` | generation model id (e.g. `claude-opus-4-8`) |
| `generated_at` | `timestamptz` | first generation time |
| `updated_at` | `timestamptz` | last (re)generation / edit time |
| `last_verified_at` | `timestamptz` | nullable — for future staleness policy |
| `edited_by` | `text` | nullable — user id of a manual edit |

- **Primary key:** `(place_id, prompt_version)` — the leading `place_id` column
  serves the per-place reads, so no extra index is needed for the hot path. (A
  small number of rows per place.)
- **RLS:** **deny all** to `anon`/`authenticated`. Only the **service role** (the
  edge function) accesses it.

### Table `place_cache_audit` (lightweight operational log)

| column | type | notes |
|---|---|---|
| `id` | `bigint` (identity) | PK |
| `place_id` | `text` | indexed |
| `prompt_version` | `int` | |
| `action` | `text` | `generate` \| `regenerate` \| `manual_edit` \| `mark_stale` |
| `actor` | `text` | user id, or `system` |
| `model` | `text` | nullable |
| `prompt_id` | `text` | nullable |
| `detail` | `text` | nullable note |
| `created_at` | `timestamptz` | default `now()` |

- Index on `(place_id, created_at)`. RLS service-role-only.
- `enrich-place` writes `generate`/`regenerate` rows. Manual dashboard edits are
  captured by a **single lightweight `AFTER UPDATE` trigger** on `place_cache`
  (one-time SQL) that appends a `manual_edit` row when content changes and
  `content_source='manual'`. A trigger is a table-level hook, not a service —
  consistent with the minimal constraint.

### Read precedence (handles manual edits + version bumps)

`get` selects the row for `place_id` preferring, in order:
1. a `content_source='manual'` row (any version) — **manual wins, version-
   independent**, so a manual edit survives version bumps and is never auto-
   regenerated over; then
2. the `prompt_version = CURRENT_ENRICH_VERSION` row.

### Stop fields (already exist from PR #1 — additive)

- `Stop.placeId?` — the cache key.
- `Stop.history/facts/tips/notice?` — **kept** but, for placeId stops, **no longer
  written by new code**; they remain a transition fallback for pre-existing stops
  and the storage for by-name stops. (Future premium offline may snapshot here.)

## Provenance & versioning

- `CURRENT_ENRICH_VERSION` constant = **2** (current "facts-separated" prompt; the
  old browser prompt is implicitly **1**). Defined once in app code (e.g.
  `enrich.ts`) for the client query key and **mirrored** in the `enrich-place`
  function (Deno can't import app modules); bumping requires updating both
  (called out in the plan).
- `prompt_id` = a stable hash/identifier of the **exact** prompt template string
  used (computed where the prompt is built, stamped on every write). So the cache
  answers *"which exact prompt produced this?"* without relying solely on the
  integer version.
- The library is read/written at `CURRENT_ENRICH_VERSION`, so old-prompt (v1)
  content is never served or stored. **Version-bump regeneration is preserved:**
  bump the integer → places regenerate at the new version on next view (except
  manual entries, which win by precedence).
- Every write also stamps `model`, `place_name`, `address`, `lat/lng`,
  `place_types`, `country_code`, `region`, `source`, and timestamps — so months
  later we know **what** place, **which** prompt + model, and **when**.

## Manual-edit protection

- A founder can hand-edit a row in the Supabase dashboard; they set
  `content_source='manual'` (and `edited_by`). The trigger logs a `manual_edit`.
- **Read precedence** makes a `manual` entry win over generated ones and ignore
  the version (above) — so manual content is served everywhere and is **never
  silently regenerated over** by a `get` miss or a version bump.
- **`regenerate` on a manual entry:** by default **preserved** (the action
  detects `content_source='manual'` and returns the manual content untouched,
  noting it). Overwriting requires explicit **`force: true`**, which flips the row
  back to `content_source='generated'` and logs a `regenerate`. So no accidental
  destruction of curated text.

## Validation & abuse protection

Before any generation, `enrich-place` validates the request — preventing malformed
or fake place ids from polluting the cache:
- `placeId` must match the Google place-id shape (`^[A-Za-z0-9_-]{16,}$`); reject
  otherwise (no row created).
- `name` required (non-empty, length-capped); `destination`/`address` length-
  capped.
- `coords` if present must be finite and in range (lat −90..90, lng −180..180).
- `placeTypes` coerced to a clean string array.

Generation abuse controls (all lazy / in-function — no workers):
- **Per-IP + per-user rate limit** on generation (`get` miss + `regenerate`),
  reusing the `place-photo` limiter pattern. `getBatch` (ready-hits only) and
  `ready` hits are not generation and aren't gated by this.
- **`generation_attempts` cap** (`MAX_ATTEMPTS`) + **`FAILURE_COOLDOWN`** stop
  repeated-failure storms (see Generation states).
- **The row lock** prevents concurrent duplicate generations for one place.
- The existing **founder/credits gate** (`ai-proxy`) still gates all generation;
  `regenerate` additionally requires founder.

## Client read layer & performance (instant, not laggy)

`useStopDescription(stop)` returns `{ history, facts, tips, notice, state }`:
- **placeId stop** → TanStack Query keyed on `['place-desc', placeId, version]`,
  `queryFn` calls `enrich-place:get`. Tuned for instant UX:
  - **In-memory cache** — fetched once per place per session, instant thereafter.
  - **Pre-warm on trip open** — a **bounded** `getBatch` (see Pre-warm scaling).
  - **Pre-warm on add** — adding a stop fires `get` in the background.
  - **Stale-while-revalidate** — shows instantly, re-checks in the background, so
    founder edits/regenerations propagate on next load without blocking.
  - On `pending`/`failed`, drive the **Loading behavior** below.
- **by-name stop** → the stop's stored `history/facts/tips` (generated once via
  `enrich-place:get` with no cache write, then saved on the stop).

Read sites updated to use this hook instead of reading `stop.history/...`:
`Guide` (`StoryTabs`), `StopDetail`, `ArrivalView`.

> Optional perf enhancement (not core): persist the query cache to localStorage
> for instant reads across reloads — a device-local perf cache, distinct from
> premium offline (synced trip-data snapshots).

## Pre-warm scaling

Large trips must not fire a request per stop. Pre-warm is **bounded**:
- Warm only the **active day's** stops (Guide) — and, in Plan, the **visible**
  stops — not every day of a long trip.
- Cap `getBatch` to `PREWARM_BATCH_MAX` placeIds (e.g. 25); if the active set
  exceeds it, warm the first batch and lazy-load the rest on view.
- `getBatch` returns **ready hits only** and never generates, so even a big warm
  is one cheap read with no AI burst.
Remaining stops fetch lazily (and instantly, once cached) when viewed.

## Loading behavior (first-gen feels intentional, never broken)

Driven by the `get` state:
- **Generating / pending** (this place is being written, by you or another user):
  a tasteful, labeled shimmer — e.g. *"Writing this place's story…"* — in the tab
  body, and the query **polls** (refetch every ~1.5s, capped at ~`LEASE_SECONDS`)
  until `ready`. Intentional, not a blank.
- **Cache miss just now claimed:** same shimmer (the claiming request is
  generating).
- **Temporary failure:** show the stop's **legacy stored fields** if present;
  otherwise a calm, non-alarming line — e.g. *"We couldn't write this place's
  story just now."* — with a single quiet retry (respecting the cooldown). No
  spinner spam, no error wall.
- **Adding/viewing a stop is never blocked** on any of the above.

## Auditability

`place_cache_audit` gives operational visibility (not an enterprise platform):
`enrich-place` appends a row on **generate** and **regenerate** (with `actor`,
`model`, `prompt_id`); the `place_cache` trigger appends on **manual_edit**; a
founder marking a row stale logs **mark_stale**. Enough to answer "who/what/when
changed this place," nothing more.

## Stale-content handling (operationally simple; no auto-regen)

- `generated_at` / `updated_at` / `last_verified_at` make age visible.
- A founder (or a future tool) can set `is_stale=true` on an entry. **Stale
  entries are still served** (no UX impact) and simply flagged for attention.
- No forced/automatic regeneration. A future time-based policy (e.g. "offer to
  refresh entries older than N years") is left open via these columns — but is
  out of scope and adds no worker now.

## Security

- `place_cache` + `place_cache_audit` are **service-role-only** (RLS denies all
  client access). The app reaches descriptions solely through `enrich-place`. **No
  user write path** → the shared library cannot be vandalized.
- **Reads of cached entries are free + ungated** (a `ready` hit does no AI work).
- **Generation** (`get` miss / `regenerate`) is **validated** (place shape/coords),
  **rate-limited** (per-IP + per-user), **lock-protected** (no duplicate calls),
  and **gated** (founder/credits via `ai-proxy`); `regenerate` requires founder.
- Google + Anthropic keys remain server-side only.

## Failure & degradation

| Condition | Behaviour |
|---|---|
| `enrich-place` / network down | hook returns the stop's legacy fields if any, else a calm empty state; **adding/viewing is never blocked** |
| Concurrent misses for one place | one generation (row lock); others get `pending` then the shared result |
| Generation fails | `status='failed'` + `generation_error`; served as `failed` with cooldown; founder can force-retry |
| Stuck `generating` (owner died) | reclaimed on next read after `LEASE_SECONDS` |
| Repeated failures | `MAX_ATTEMPTS` cap → only founder `regenerate` retries |
| Invalid/fake placeId | rejected by validation — no row created |
| Pre-warm batch fails | ignored — descriptions fetch lazily on view |

## Cost

- Library hits eliminate repeat AI calls (the core saving) and are free/ungated.
- Generation only on a true miss or founder `regenerate`, gated + rate-limited +
  lock-deduplicated. `getBatch` never generates, so opening a trip triggers no AI
  burst.

## Acceptance criteria

- A place's description is generated **once** and reused across trips/users (a
  second add of the same `placeId` is a free, ungated library hit).
- **Concurrent** misses for the same place trigger **exactly one** AI generation;
  others receive `pending` then the shared result.
- A **stuck `generating`** row is reclaimed on the next read after the lease;
  repeated failures are capped and cooled down, not retried forever.
- Editing or regenerating a place updates **every** stop of that place across
  trips (on next load); a **manually-edited** entry is **served everywhere**,
  **survives version bumps**, and is **not overwritten** by a `get` or a
  non-`force` `regenerate`.
- The `place_cache`/`place_cache_audit` tables are **not readable or writable by
  client roles** — only `enrich-place` (service role). No per-user write path.
- **Invalid/fake placeIds are rejected** before any row is created; generation is
  **rate-limited** per IP + user and stays behind the **founder/credits gate**.
- Reads are **instant** after pre-warm/first fetch; pre-warm is **bounded**
  (active-day, batch cap) and triggers **no** generation.
- First-generation, pending, and failure states render **intentional** UX and
  **never block** adding/viewing a stop.
- Every cache entry records **place name/address/coords/types**, **model**,
  **prompt_id** + version, and **generated/updated** timestamps.
- **generate / regenerate / manual_edit / mark_stale** are recorded in the audit
  log with actor + timestamp.
- Old-prompt (v1) content is never served or written; version-bump regeneration
  still works for generated (non-manual) entries.
- Typed-by-name stops still get a description (stored on the stop), without the
  library.
- No background jobs, queues, cron, realtime, or new services are introduced.

## Out of scope / future

- **Premium offline:** snapshot descriptions into trip data for synced offline.
- **Admin "Places" page:** search a place → view/edit/regenerate/mark-stale.
- **Realtime push** of edits to open clients (vs refetch-on-load).
- **Time-based staleness policy** (auto-offer refresh past N years) — enabled by
  the timestamp/`is_stale` columns, built later.
- **Shared cache for by-name stops** (needs a fuzzy name/coords key).
