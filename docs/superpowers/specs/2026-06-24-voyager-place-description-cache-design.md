# Shared Place-Description Cache ‚Äî Design Spec

> Status: approved (brainstorm) ¬∑ 2026-06-24 ¬∑ branch `enrichment-cache`
> Builds on the merged `placeId` work (PR #1): stops carry a canonical Google
> `placeId`, which is the key this cache is built on.

## Context & goal

Today a stop's description ‚Äî **Story (`history`)**, **Interesting Facts
(`facts`)**, **Experience (`tips`)** (plus legacy `notice`) ‚Äî is generated by the
AI **per stop** (`generateStopDetail` runs in the browser) and stored inline on
the stop. The same place in two trips generates twice; a deleted-and-re-added
place regenerates; and there is no single place to maintain a description.

We want a **global, shared library of descriptions** keyed by Google `placeId`:
generate a place's description **once**, reuse it everywhere, and let an admin
**maintain** it (edit/regenerate once ‚Üí it updates every stop of that place in
every trip). Reads must be **instant** (no per-view lag). The library must be
**sealed** ‚Äî no user can write arbitrary text into it.

## Core model ‚Äî one shared, server-controlled source of truth

- A place's description lives in **one** global library entry (`place_cache`),
  keyed by `placeId` + `prompt_version`. **It is not copied onto the stop** (for
  placeId stops).
- A stop with a `placeId` **displays whatever the library currently holds** for
  that place. Change the library entry once ‚Üí every stop of that place, in every
  trip, shows the new text (on its next load ‚Äî see Performance).
- The library is **server-only**: the app never reads or writes the table
  directly. A single **edge function (`enrich-place`)** is the only thing that
  touches it (service role). So there is **no client write path** ‚Äî a signed-in
  user cannot inject "hey jeorgjeorgi".
- **Typed-by-name stops** (no `placeId`) have no shared identity, so they keep
  their own per-stop copy (today's behavior), generated once and stored on the
  stop.

## Non-goals

- **Offline trips are out of scope** (deferred to a future **premium** tier):
  free tier reads descriptions live (cached for speed). Premium will later
  snapshot descriptions into trip data for true cross-device offline. This spec
  leaves room for it but does not build it.
- **No per-user "Generate / Re-generate" button** ‚Äî regeneration is admin-only
  (see Maintenance). Removes spam/cost-abuse.
- **No per-stop description storage for placeId stops** ‚Äî they read from the
  library. (Existing stored fields remain only as a transition fallback.)
- Not building a full admin "Places" management UI in v1 (a minimal admin
  regenerate action suffices; the dashboard covers hand-edits).
- No realtime push in v1 (refetch-on-load propagates edits; realtime is a future
  enhancement).

## Architecture & data flow

```
DISPLAY a stop's description
  useStopDescription(stop):
    if stop.placeId ‚Üí read library via enrich-place (cached client-side; see Performance)
                       hit  ‚Üí show it
                       miss ‚Üí enrich-place generates (Wikipedia + AI, gated),
                              writes the library, returns it ‚Üí show it
                       (while a first-gen is in flight, fall back to the stop's
                        legacy stored fields if present, else a skeleton)
    else (by-name)  ‚Üí show the stop's own stored history/facts/tips
                       (generated once via enrich-place with no cache write)

GENERATE / WRITE (only inside enrich-place, service role)
  Wikipedia grounding + buildEnrichPrompt + AI (via ai-proxy, founder/credits-gated)
  + parseStopDetail ‚Üí write place_cache[placeId @ current version] ‚Üí return
```

### The edge function: `supabase/functions/enrich-place`

The single authority for reading, generating, and writing descriptions. The
existing browser-side `generateStopDetail` pipeline (Wikipedia extract ‚Üí
`buildEnrichPrompt` ‚Üí AI ‚Üí `parseStopDetail`) **moves server-side** into this
function (the pure helpers are ported to Deno). Actions:

- **`get`** `{ placeId, name, destination?, coords? }` ‚Üí returns
  `{ history, facts, tips, notice }`:
  - Read `place_cache` by `(placeId, CURRENT_VERSION)` with the service key.
  - **Hit** ‚Üí return it (no AI, **free + ungated** ‚Äî viewing cached places costs
    nothing).
  - **Miss** ‚Üí enforce the founder/credits gate (reuse `ai-proxy`), generate
    (Wikipedia + AI), write the library at `CURRENT_VERSION`, return it.
- **`getBatch`** `{ placeIds: [...] }` ‚Üí returns a `{ placeId ‚Üí description }` map
  of **cache hits only** (no generation) ‚Äî used to pre-warm a trip on open in one
  round trip. Misses are generated lazily (on view / on add).
- **`regenerate`** `{ placeId, name, destination?, coords? }` ‚Üí **founder-only**
  ("admin" = the existing `profile.role === 'founder'`; the server verifies it):
  force a fresh generation, overwrite the library entry, return it. Backs the
  admin Regenerate action.

The function never returns the Google/Anthropic keys. Graceful: any failure (key
unset, AI down, bad input) returns an empty/sentinel result, never a 5xx ‚Äî the UI
degrades to the stop's existing text or empty.

> **Server-side test caveat:** Deno isn't available locally, so the function's
> pure helpers (ported `buildEnrichPrompt`/`parseStopDetail`/version logic) are
> unit-tested in the Node/vitest app where they're authored and shared if
> feasible; the deployed function is verified by a live probe. The client read
> layer + caching + hook logic are fully unit-tested in vitest.

## Data model

### New table `place_cache` (global; operator-created SQL ‚Äî no migration framework in repo)

| column | type | notes |
|---|---|---|
| `place_id` | `text` | Google place id |
| `prompt_version` | `int` | which prompt produced this (current = 2) |
| `history` | `text` | Story |
| `facts` | `jsonb` | Interesting Facts (string array) |
| `tips` | `text` | Experience |
| `notice` | `text` | legacy field, nullable |
| `source` | `text` | optional: the Wikipedia extract used (provenance) |
| `generated_at` | `timestamptz` | first generation time |
| `updated_at` | `timestamptz` | last (re)generation / edit time ‚Äî **staleness visibility** |

- **Primary key:** `(place_id, prompt_version)` ‚Äî multiple versions can coexist;
  reads/writes target `CURRENT_VERSION`.
- **RLS:** **deny all** to `anon`/`authenticated` (no client SELECT/INSERT/
  UPDATE/DELETE). Only the **service role** (the edge function) accesses it. (The
  app reaches it solely through `enrich-place`.)

### Stop fields (already exist from PR #1 ‚Äî additive)

- `Stop.placeId?` ‚Äî the cache key (present on autocomplete-added stops).
- `Stop.history/facts/tips/notice?` ‚Äî **kept in the type** but, for placeId stops,
  **no longer written by new code**; they remain a **transition fallback** for
  pre-existing stops and the storage for **by-name** stops. (A future premium
  offline tier may write snapshots here.)

### Versioning

- `CURRENT_ENRICH_VERSION` constant = **2** ‚Äî the current "facts-separated /
  experience-routed" prompt. The pre-existing browser prompt is implicitly version
  **1**. Defined once in app code (e.g. `enrich.ts`) for the client query key, and
  **mirrored** in the `enrich-place` function (same integer) since Deno can't
  import app modules; bumping it requires updating both (called out in the plan).
- The library is **only ever read/written at `CURRENT_ENRICH_VERSION`**, so
  old-prompt output (version 1) is never served or stored. Bumping the constant
  (when the prompt improves) makes every place regenerate at the new version on
  next view.

## Client read layer & performance (instant, not laggy)

`useStopDescription(stop)` returns `{ history, facts, tips, notice }`:

- **placeId stop** ‚Üí a TanStack Query keyed on `['place-desc', placeId, version]`,
  `queryFn` calls `enrich-place:get`. Behavior tuned for instant UX:
  - **In-memory cache** ‚Äî fetched once per place per session, instant thereafter.
  - **Pre-warm on trip open** ‚Äî one batched `getBatch` for all the trip's placeId
    stops fills the cache up front, so tapping into any stop is instant.
  - **Pre-warm on add** ‚Äî adding a stop fires `get` in the background so it's
    ready before the user opens it.
  - **Stale-while-revalidate** ‚Äî a reasonable `staleTime`; the cache shows
    instantly and re-checks in the background, so admin edits/regenerations
    propagate on the next load without ever blocking the view.
  - **First-gen fallback** ‚Äî while a cache miss is generating, show the stop's
    legacy stored fields if present, else a skeleton.
- **by-name stop** ‚Üí read the stop's stored `history/facts/tips` (generated once
  via `enrich-place:get` with no cache write, then saved onto the stop).

Read sites updated to use this hook instead of reading `stop.history/...`
directly: `Guide` (`StoryTabs`), `StopDetail`, `ArrivalView`.

> Optional perf enhancement (not core): persist the query cache to localStorage
> for instant reads across reloads. Kept distinct from premium offline (which is
> synced trip-data snapshots, not a device-local perf cache).

## Auto-enrichment (no button)

- **On add** (AddStop ‚Äî autocomplete, AI-suggested, or by-name): fire
  `enrich-place:get` in the background to pre-warm the description (library for
  placeId stops; stored-on-stop for by-name).
- **On view**: `useStopDescription` fetches/generates as needed (covers
  pre-existing stops that were never pre-warmed).
- Guide's current `!stop.history` auto-enrich and StopDetail's button are
  **replaced** by this hook + the admin regenerate.

## Maintenance (admin-only)

Three levers, none exposed to regular users:

1. **Regenerate one place** ‚Äî an **admin/founder-only** action (UI control or a
   minimal admin surface) calling `enrich-place:regenerate`. Forces a fresh
   generation, overwrites the library entry ‚Üí every stop of that place updates.
2. **Refresh everything (version bump)** ‚Äî bump `CURRENT_ENRICH_VERSION` when the
   prompt improves; places regenerate at the new version as they're viewed (or a
   batch pre-warm warms them). This is the "bring it all up to par" tool.
3. **Hand-edit** ‚Äî edit the `place_cache` row directly in the Supabase dashboard
   (or SQL) for a one-off manual fix; `updated_at` records when.

## Security

- The `place_cache` table is **server-role-only** (RLS denies all client access).
  The app reaches descriptions exclusively through `enrich-place`. **No user write
  path exists** ‚Üí the shared library cannot be vandalized.
- **Reads of cached entries are free + ungated** (a library hit does no AI work).
- **Generation (cache miss / regenerate)** flows through the existing
  **founder/credits gate** (`ai-proxy`), so AI cost stays controlled; `regenerate`
  additionally requires admin/founder.
- Google + Anthropic keys remain server-side only.

## Failure & degradation

| Condition | Behaviour |
|---|---|
| `enrich-place` / network down | `useStopDescription` returns the stop's legacy stored fields if any, else empty; **adding/viewing a stop is never blocked** |
| Cache miss + AI down/ungated | return empty (no fabricated text); the place shows no description until a later successful generation |
| Bad/garbage AI output | `parseStopDetail` guards (existing); empty rather than malformed; not written |
| Pre-warm batch fails | silently ignored ‚Äî descriptions fetch lazily on view |

## Cost

- Reuse (library hits) eliminates repeat AI calls ‚Äî the core saving.
- Generation is gated (founder/credits) and only happens on a true miss or an
  admin regenerate. Pre-warm `getBatch` does **not** generate (hits only), so
  opening a trip never triggers a burst of AI calls.

## Acceptance criteria

- A place's description is generated **once** and reused across trips/users
  (second add of the same `placeId` does no AI work ‚Äî a library hit).
- Editing or regenerating a place's library entry updates **every** stop of that
  place across trips (on next load).
- The `place_cache` table is **not writable or readable by client roles** ‚Äî only
  the `enrich-place` function (service role). No per-user write path.
- Reads are **instant** after pre-warm/first fetch (no per-view network lag);
  opening a trip triggers at most one batched, generation-free pre-warm.
- **No per-user regenerate button**; regeneration is **admin-only**.
- Old-prompt (version 1) content is never served or written to the library; the
  current version (2) is authoritative; bumping the version regenerates on view.
- Typed-by-name stops still get a description (stored on the stop), without the
  library.
- Generation stays behind the founder/credits gate; cached reads are free.
- No path blocks adding/viewing a stop on enrichment.

## Out of scope / future

- **Premium offline:** snapshot descriptions into trip data for synced offline.
- **Admin "Places" page:** search a place ‚Üí view/edit/regenerate.
- **Realtime push:** instant propagation of edits to open clients (vs refetch).
- **Time-based staleness policy:** auto-refresh entries older than N years
  (the `generated_at`/`updated_at` columns make this possible later).
- **Shared cache for by-name stops** (would need a fuzzy name/coords key).
